OpenHarness Fork Architecture

Evidence-backed system map of the interactive harness, YAML execution layer, Harbor evaluation path, and ohmo personal-agent gateway.

Core Components

  • • `openharness.cli` launches interactive, print, backend-only, setup, auth, provider, MCP, plugin, and cron paths.
  • • `build_runtime` assembles settings, auth, MCP, plugins, hooks, commands, tools, app state, QueryEngine, RunContext, and tracing.
  • • `AgentRuntime` reuses the same query/tool primitives for YAML-configured workflow agents.
  • • `ohmo` layers a personal workspace, session backend, channel gateway, and extra skills/plugins on the shared runtime.

Data Flow

  • • React Ink speaks JSON lines to a Python backend host, which emits transcript, tool, status, task, MCP, and bridge snapshots.
  • • Query loops call provider clients, dispatch approved tools, stream events, and persist message/event logs through RunContext.
  • • YAML catalogs feed AgentFactory, which builds simple, planner/executor, reflection, or ReAct agents over AgentRuntime.
  • • Chat channel messages enter `MessageBus`, route through `OhmoGatewayBridge`, and reuse per-session runtime bundles.

Deployment / Security

  • • This repo primarily describes local host processes, subprocesses, Docker sandboxes, and Harbor jobs, not a cloud deployment.
  • • PermissionChecker, tool allow/deny settings, path rules, and optional Docker sandboxing sit between tools and the workspace.
  • • Secrets are resolved from provider profiles, environment variables, or external CLI bindings; values are not embedded here.
  • • Langfuse traces and `runs/<run_id>/` artifacts provide the reproducibility and observability record.
